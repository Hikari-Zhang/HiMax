### 2. Cookie 系统三个 Bug 修复
- **Bug 1**: `_browserCookieToNetscape` 没有清理引号包裹的 cookie 字符串，导致 cookie name/value 带有多余引号（如 `'__ac_signature` 和 `true'`）
  - 修复：增加首尾引号去除 + name/value 引号清理
- **Bug 2**: Cookie 过期时间写死为 `0`（session cookie），yt-dlp 可能忽略
  - 修复：改为当前时间 + 1 年
- **Bug 3**: 抖音只写入 `.douyin.com` 域的 cookie，但 yt-dlp 解析抖音时也需要 `.iesdouyin.com` 域的 cookie（如 ttwid）
  - 修复：SITE_PRESETS 增加 `extraDomains` 字段，saveSiteCookies/deleteSiteCookies 同时处理多域
- **额外**: 增加 `_sanitizeCookieFile` 方法，在 `_buildCookieArgs` 中自动清理已有的脏数据
- secure 标志从 FALSE 改为 TRUE（HTTPS 站点）

### 3. 抖音 cookie 根因分析（深度排查）
- **根因确认**：`document.cookie` 无法获取 httpOnly cookie（如 `msToken`），这是抖音反爬的关键 cookie
- 直接用 yt-dlp 命令行 + cookies.txt 文件测试也失败 → 证明不是代码传递问题，是 cookie 内容不完整
- `--cookies-from-browser edge` 也失败 → Edge 运行中，数据库被锁（"Could not copy Chrome cookie database"）
- yt-dlp 更新到 nightly 2026.03.21.233500 也无法解决 → 确认是 cookie 不完整的问题
- **解决方案**：必须用浏览器扩展（如 "Get cookies.txt LOCALLY"）导出完整 cookies.txt（包含 httpOnly），然后导入
- 已更新两处抖音错误提示，改为推荐浏览器扩展导出方式，不再建议 document.cookie

### 4. 自定义抖音提取器（重大功能）
- **根因升级**：yt-dlp 对抖音的 extractor 本身就有 bug（GitHub issue #9667，2024年4月至今未修复），即使提供完整 cookie 也报 "Fresh cookies needed"
- **解决方案**：实现了 `electron/services/douyin-extractor.cjs`，完全不依赖 yt-dlp，不需要 cookie
- **原理**：
  1. 访问 `https://www.iesdouyin.com/share/video/{video_id}/`（模拟移动端 Android UA）
  2. 从 HTML 中正则提取 `window._ROUTER_DATA` JSON 数据
  3. 解析得到视频标题、作者、时长、1080p 无水印下载 URL
  4. 直接 HTTP 下载视频文件（跟随 302 重定向到 CDN）
- **集成方式**：
  - `ytdlp-engine.cjs` 构造函数初始化 `this.douyinExtractor`
  - `parseUrl` 中抖音 authError（无论有无 cookie）→ 自动 fallback 到自定义提取器
  - `download-manager.cjs` 中 task 携带 `_douyinDirect`/`_douyinVideoUrl` 字段
  - `_startTask` 检测到 `_douyinDirect` 时调用 `_startDouyinDirectDownload` 而非 yt-dlp
- **测试结果**：91.7MB 视频完整下载，1080p 无水印，不需要任何 cookie
- **注意**：抖音 API 可能会更新，`_ROUTER_DATA` 结构如果变化需要调整正则

### 13. 快捷键系统三个 Bug 修复
- **Bug 1（根因）**：`main.cjs` 中 `shortcutManager` 以 `null` 值传入 `registerIpcHandlers`（JS 值传递），IPC handlers 闭包中的 `shortcutManager` 永远是 `null` → `shortcut:update`/`reset-all` 始终返回 `NOT_INITIALIZED` 错误
  - 修复：将 `shortcutManager` 初始化移到 `registerIpcHandlers` 之前
- **Bug 2（致命缺失）**：前端完全没有注册 `onShortcutTriggered` 监听器，即使后端拦截了快捷键并发送 IPC 事件，渲染进程也没有任何代码处理
  - 修复：在 `App.tsx` 中 `useEffect` 注册监听器，处理所有快捷键动作：
    - `pasteAndDownload`：检查焦点不在输入框 → 读取剪贴板 → 提取 URL → 调用 `parseUrl` 自动解析
    - `pauseAll`/`resumeAll`：调用对应 electronAPI
    - `openSettings`：切换到设置页面
    - `toggleClipboard`：切换剪贴板监控设置
- **Bug 3（重复注册）**：`_registerInApp()` 每次 `registerAll()` 时都会新增一个 `before-input-event` 监听器，但从不移除旧的 → 快捷键会被重复触发 N 次
  - 修复：添加 `_inAppRegistered` 标志位，确保只注册一次（回调动态读取 `this.shortcuts`）
- **优化**：`pasteAndDownload` (Ctrl+V) 的 `before-input-event` 不再调用 `event.preventDefault()`，允许正常的浏览器粘贴行为继续生效；前端监听器判断焦点在输入框时跳过快捷键动作

### 23. StatusBar 速度显示 0.0 MB/s 修复
- **根因**：yt-dlp 输出速度格式为 `5.20MiB/s`（二进制单位、无空格），StatusBar 正则只匹配 `MB/KB/GB`（十进制单位、需空格）
- **修复**：正则改为 `/([\d.]+)\s*(GiB|GB|MiB|MB|KiB|KB|B)(?:\/s)?/i`，支持所有单位格式
- 两条速度来源：yt-dlp 引擎直传原始值（`MiB/s`）、抖音自定义下载器用 `_formatSpeed` 格式化（`MB/s`）

### 24. 下载中"打开文件夹"按钮无效修复
- **根因**：任务创建时 `filepath: null`，yt-dlp 需要输出 `[download] Destination:` 行后才设置 filepath。下载初期 filepath 为空，`if (task.filepath) showInFolder(...)` 静默跳过无反馈
- **修复**（三处）：
  1. `TaskCard.tsx`：`showInFolder(task.filepath || task.outputDir)` — filepath 为空时 fallback 到 outputDir
  2. `CompletedPage.tsx`：同样 fallback 到 outputDir
  3. `ipc-handlers.cjs` `shell:show-in-folder`：新增目录检测，如果传入的是目录路径则 `shell.openPath()` 直接打开目录（而不是 `showItemInFolder` 高亮文件）

### 25. 下载中关闭程序后无法继续下载的修复
- **根因**：`_loadTasks()` 正确地将 `downloading` → `queued`，但构造函数中没有调用 `_processQueue()`，queued 任务永远不会自动开始
- **修复**（三处）：
  1. `download-manager.cjs` 构造函数：`_loadTasks()` 后检测 queued 任务数量，如果 > 0 则 setTimeout 3s 后自动 `_processQueue()`（给应用完成初始化的时间）
  2. `download-manager.cjs` `resumeTask()`：扩展支持 `queued` / `download_failed` / `parse_failed` 状态（原来只支持 `paused`），让用户也能手动恢复重启后的任务
  3. `TaskCard.tsx` queued 状态 UI 改进：显示 "等待中" 文案 + 之前的进度百分比（如有）+ 已下载/总大小
- i18n 新增 `downloading.queued`：中文"等待中"、英文"Queued"
- yt-dlp 本身支持断点续传（默认 `--continue`），重新运行时自动续传 `.part` 文件

### 26. 偶尔文件下载好了但UI不显示完成的修复
- **可能根因**：
  1. `_notifyListeners` 传递包含 `_downloadOptions` 等内部字段的原始 task 对象，IPC 序列化可能静默失败导致 `task-updated` 丢失
  2. `loadTasks()` 被高频事件反复触发且无防抖，IPC 响应乱序可能导致旧数据覆盖新数据
  3. yt-dlp 进程 `close` 事件在极端情况下可能不触发（Node.js 已知边缘问题）
- **修复**（4处）：
  1. `download-manager.cjs` 新增 `_cleanTaskForIPC()` 方法，`_notifyListeners` 对 `task-updated`/`task-added` 事件自动清理内部字段后再发送
  2. `download-manager.cjs` `complete` handler：增加详细日志 + **1秒后重发一次 `task-updated` 通知**作为安全网
  3. `download-manager.cjs` 新增 `_startWatchdog()` 看门狗：每10秒检查 yt-dlp 进程是否仍活跃，如果进程已退出但任务仍处于 downloading/post_processing 状态，强制完成或标记失败（检测输出文件是否存在来决定）
  4. `store/index.ts` 前端事件处理重构：
     - `loadTasks`/`loadStats` 加150ms/500ms防抖，避免快速连续的 IPC 请求竞态
     - `task-updated` 事件增加**即时局部更新**：收到完成通知立即将任务从 downloading 移到 completed，不等 IPC 响应
     - `task-progress` 扩展支持 `post_processing` 状态（之前只更新 `downloading`）

### 28. 数据存储路径改到安装目录（卸载干净）
- **新增** `electron/services/app-paths.cjs` — 统一路径管理模块
- **目录结构**：所有应用数据存放在 `<安装目录>/appdata/` 下
  - `appdata/bin/` — yt-dlp、ffmpeg、ffprobe
  - `appdata/data/` — settings.json、tasks.json
  - `appdata/debug/` — parse_*.json（调试用）
  - `appdata/cookies.txt` — Netscape cookie 文件
- **之前**：所有数据在 `%APPDATA%\Downie`（`app.getPath('userData')`），卸载后残留
- **现在**：所有数据在安装目录内，卸载时随目录一起删除
- **路径解析**：
  - 开发模式：`app.getAppPath()`（项目根目录）
  - 打包后：`path.dirname(process.resourcesPath)`（安装目录）
- **修改文件**：
  - `electron/main.cjs` — 引入 app-paths，启动时 ensureDir
  - `electron/services/binary-manager.cjs` — binDir 改用 `getBinDir()`
  - `electron/services/download-manager.cjs` — dataDir 改用 `getDataDir()`
  - `electron/services/ytdlp-engine.cjs` — cookie/debug 路径改用 `getCookiePath()`/`getDebugDir()`
- **保留** `app.getPath('downloads')` 引用（用户下载目录，不属于应用数据）
