## B站视频解析信息错乱修复
- 问题：用户反馈解析 B 站视频后所有信息都是乱的
- 根因分析：B 站使用 DASH 格式，音视频流分离，与 YouTube 结构差异大
  - ext 是 `m4s` 不是 `mp4`
  - 视频流没有 `resolution` 字段，只有 `height`/`width`
  - codec 字段是原始格式（如 `avc1.640028`），不友好
  - `format_note` 是中文（如"高清 1080P"）
  - 缩略图可能 `//` 开头缺少协议
  - duration 可能是毫秒级
- 修复内容：
  - `electron/services/ytdlp-engine.cjs` — 重写 `_normalizeVideoInfo()`：
    - 增强 resolution 提取（优先用 height，回退 resolution 字段）
    - m4s 扩展名显示为 mp4
    - codec 标准化（avc→H.264, hev→H.265, av01→AV1, vp9→VP9）
    - 按分辨率去重，取最佳格式
    - HDR/fps 标注
    - 缩略图协议修复
    - duration 毫秒检测
    - 添加 debug JSON 保存（%APPDATA%/Downie/debug/）
  - 下载格式选择逻辑修复：始终使用 `bestvideo+bestaudio` 模式
  - `VideoPreviewModal.tsx` — 添加 `referrerPolicy="no-referrer"` 修复 B 站缩略图

## 真正的根因：DPAPI Cookie 解密失败
- 症状：用户反馈解析结果是 mock 数据，实际上是解析失败后 parsedVideoInfo=null 导致 VideoPreviewModal 回退到 mock
- 真正原因：用户在设置里开启了 cookie browser 模式，但 yt-dlp 在 Windows 上用 DPAPI 解密 Edge cookie 时失败
  - 错误信息：`ERROR: Failed to decrypt with DPAPI`
  - 参考：https://github.com/yt-dlp/yt-dlp/issues/10927
  - Electron 自身就是 Chromium 内核，可能干扰 DPAPI 解密
- 修复：`ytdlp-engine.cjs` — `parseUrl()` 重构为 async + `_runParse()` 分离
  - 第一次尝试带 cookie 解析
  - 如果报 DPAPI/decrypt/cookie 相关错误，自动 fallback 到无 cookie 重试
  - 成功时附带 `_cookieWarning` 字段提示用户
  - 两次都失败则合并错误信息抛出

### 6. filepath 中文乱码修复（GBK → UTF-8）
- **根因**：Windows 系统默认控制台代码页是 GBK (CP936)，yt-dlp（Python 程序）的 stdout 默认用系统代码页输出。Node.js `data.toString()` 默认按 UTF-8 解码 → 中文变乱码
- **修复方案**：在所有 spawn yt-dlp 的调用中设置 `PYTHONIOENCODING=utf-8` 和 `PYTHONUTF8=1` 环境变量
- **`ytdlp-engine.cjs` 改动**：
  - 新增 `_buildUtf8Env()` 方法：返回 `{ ...process.env, PYTHONIOENCODING: 'utf-8', PYTHONUTF8: '1' }`
  - `_buildCleanEnv()` 也加入了这两个变量
  - 4 处 yt-dlp spawn 调用（parseUrl、parsePlaylist、startDownload、exportCookies）都加了 `env` 参数
- **`download-manager.cjs` 改动**：
  - `_loadTasks()` 增加旧任务 filepath 修复：如果 completed 任务的 filepath 文件不存在，自动调用 `_findOutputFile()` 尝试重新定位
  - 修复后自动 `_saveTasks()` 持久化
- **关键点**：tasklist 的 spawn 不需要 Python 环境变量（只检测进程名，全英文）

### 7. 暂停功能失效修复
- **现象**：点暂停按钮后，任务短暂显示 paused 状态，然后又变回 downloading 继续下载
- **Bug 1 — 后端 progress handler 覆盖状态**：
  - `_setupEngineEvents` 的 progress 处理中有 `if (task.status !== 'downloading') { task.status = 'downloading' }`
  - yt-dlp 进程被 SIGTERM 杀死前 stdout buffer 中还有 progress 数据，这些数据到达后会把 paused 改回 downloading
  - **修复**：progress handler 开头加 `if (task.status !== 'downloading') return` 直接忽略
- **Bug 2 — 前端 task-progress 强制设状态**：
  - `onDownloadEvent` 中 `task-progress` case 每次都设 `status: 'downloading' as TaskStatus`
  - **修复**：只更新 `t.status === 'downloading'` 的任务，不再强制设 status
- **Bug 3 — close 事件逻辑错误**：
  - `ytdlp-engine.cjs` 第 1195 行先 `activeProcesses.delete(taskId)`，然后 1201 行 `activeProcesses.get(taskId)` 永远是 undefined
  - **修复**：先 `has()` 检查，再 `delete()`，用 `wasTracked` 变量区分是正常退出还是被 pause/cancel 杀死
- **额外保护**：
  - `postprocess` handler 只在 `task.status === 'downloading'` 时才转为 post_processing
  - `complete` handler 只在 `downloading/post_processing` 状态时才处理
  - `error` handler 忽略 paused/cancelled 状态的任务

### 8. Cookie 认证错误提示优化
- **现象**：YouTube 配置了 cookie 后还是解析失败，报错 "Sign in to confirm you're not a bot"
- **根因**：用户的 cookies.txt 只包含访客级 cookie（VISITOR_INFO1_LIVE, PREF 等），缺少 Google 登录态 cookie（SID, HSID, SSID, __Secure-1PSID 等）。yt-dlp 带着 cookie 运行但 YouTube 不认。
- **原代码问题**：
  - `parseUrl` 的 `isAuthError` 检查包含 `errMsg.includes('cookies')`，匹配太宽泛
  - 不管用户有没有配置 cookie，错误信息都一样是 "Go to Settings > Cookie > Add Cookie by Site"
  - 前端 `UrlInputBar` 的错误显示只有 150px 宽 + truncate，详细信息看不到
- **修复**：
  - 收紧 `isAuthError` 匹配条件：`cookies` 必须和 `authentication`/`login` 同时出现
  - 区分两种情况：有 cookie 但不够 vs 完全没 cookie，给出不同的错误信息
  - 有 cookie 但不够时：按网站给出具体的修复指引（YouTube 要登录 cookie，B 站要 SESSDATA 等）
  - 增强日志：在 parseUrl 开始时打印使用的 cookie 策略和文件状态
  - 前端改进：错误提示可点击展开为完整面板，支持多行 pre 格式显示

### 9. YouTube n-parameter challenge 修复
- **现象**：YouTube URL 解析报 "Requested format is not available"，`--list-formats` 只显示 storyboard 图片
- **根因**：yt-dlp 2026.03.17 需要外部 JS 运行时来解密 YouTube 的 n-parameter（反爬签名），默认只启用 deno，但系统没装 deno。虽然有 Node.js v24.14.0，但 yt-dlp 不会自动使用非 deno 运行时

  - 在 `_runParse`、`parsePlaylist`、`startDownload` 三处的 args 构建中都加入了 `...this._buildCommonArgs()`
  - 新增 n-challenge 错误检测：匹配 "Requested format is not available" / "n challenge solving failed" / "Only images are available"，给出安装 Node.js 的提示
- **验证**：`--js-runtimes node` 后，`[jsc:node] Solving JS challenges using node` 成功，完整视频格式列表正常显示

### 10. GBK 编码乱码终极修复（filepath 不可用）
- **现象**：下载完成的中文视频文件名变成乱码（`\u{FFFD}` 替换字符），`shell.openPath` 打开失败
- **根因分析**：
  - `_buildUtf8Env()` 设置了 `PYTHONIOENCODING=utf-8`，但 yt-dlp 的 PyInstaller 打包 exe 可能不完全响应该环境变量
  - 系统代码页是 936 (GBK)，yt-dlp 输出的文件路径可能仍是 GBK 编码
  - Node.js `data.toString()` 默认按 UTF-8 解码 GBK 字节流 → 产生 `\uFFFD` 替换字符
- **修复方案（多层防御）**：
  1. **`_decodeOutput(buffer)` 方法**（ytdlp-engine.cjs）：先 UTF-8 解码，检测到 `\uFFFD` 时自动回退 GBK 解码（使用 Node.js 内置 `TextDecoder('gbk')`）
  2. 在 `startDownload`、`_runParse`、`parsePlaylist`、`exportCookies` 四处 stdout/stderr 处理中全部替换为 `_decodeOutput()`
  3. `_runParse` 和 `parsePlaylist` 改为先收集 `Buffer[]`，最终 `Buffer.concat()` 后统一解码（避免跨 chunk 拆分导致乱码）
  4. **download-manager complete 事件**：增强修复逻辑，不仅处理 `filepath === null`，也处理 `filepath` 存在但 `fs.existsSync()` 为 false 的情况（乱码路径）
  5. **ipc-handlers shell:open-path / shell:show-in-folder**：文件不存在时，通过 `_findOutputFile()` 智能查找正确文件路径
- **已确认**：Node.js v24 内置 `TextDecoder('gbk')` 可用，GBK 字节 `[0xc4,0xe3,0xba,0xc3]` 正确解码为 "你好"

### 11. 暂停功能实际不停止下载
- **现象**：点击暂停后 UI 正确显示 paused 状态，但 yt-dlp 实际还在继续下载
- **根因**：`pauseDownload()` 使用 `process.kill('SIGTERM')` 杀死进程。Windows 上虽然能终止 yt-dlp 主进程，但 **yt-dlp 会 spawn ffmpeg 等子进程**，`kill('SIGTERM')` 不会杀死子进程树，导致下载继续
- **修复**：
  - 新增 `_killProcessTree(proc)` 方法
  - Windows：使用 `execSync('taskkill /T /F /PID <pid>')` 杀死整个进程树（/T = tree, /F = force）
  - macOS/Linux：使用 `process.kill(-pid, 'SIGTERM')` 杀死进程组
  - `cancelDownload()`、`pauseDownload()`、`killAll()` 三处全部改用 `_killProcessTree()`
  - 先 `activeProcesses.delete()` 再 `_killProcessTree()`（保证 close 事件中 wasTracked=false → emit 'cancelled'）
  - 添加了 `execSync` 导入
